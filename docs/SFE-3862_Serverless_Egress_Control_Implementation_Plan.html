1. Background: Serverless Egress Controls

What are Serverless Egress Controls?

Serverless egress control is a Databricks security feature that allows account admins to manage outbound network connections from serverless compute resources, reducing the risk of data exfiltration.

Supported Serverless Products:

  • Notebooks (serverless compute)
  • Workflows (serverless jobs)
  • SQL warehouses (serverless SQL)
  • Lakeflow Spark Declarative Pipelines
  • Mosaic AI Model Serving
  • Data Quality Monitoring
  • Databricks Apps (limited support)

Network Policy Modes:

Mode Description Security Level
Full Access Unrestricted outbound access to internet and network resources ‚ùå Low Security
Restricted Access Deny-by-default policy with granular control over internet, cloud storage, and Databricks API connections ‚úÖ High Security

Key Security Properties:

  • Account-level configuration (not workspace-level)
  • 1:N relationship - Applied to multiple workspaces
  • Each workspace can only have ONE network policy
  • Maximum 2500 destinations per policy
  • Maximum 100 FQDNs as allowed domains
  • Enforcement modes: ENFORCED or DRY_RUN
  • Violations logged in system.access.outbound_network table
  • Availability: AWS Enterprise tier, Azure Premium tier

2. Proposed Security Checks

NS-9: Serverless Workspaces Have Network Policies Configured

Severity: High | Check ID: 111 | Category: Network Security

Description: Validates that workspaces using serverless compute have network policies configured to control outbound connections.

Logic:

  1. Query workspaces with serverless compute enabled
  2. Check if a network policy is assigned
  3. Report workspaces using serverless without network policies

Recommendation: Configure network policies for all workspaces using serverless compute to control outbound connections and reduce data exfiltration risk.

Cloud Support: AWS ‚úÖ | Azure ‚úÖ | GCP ‚ùå

NS-10: Network Policies Use Restricted Access Mode

Severity: High | Check ID: 112 | Category: Network Security

Description: Ensures network policies use restricted access mode instead of full access.

Logic:

  1. Query all account network policies
  2. Identify policies with access_mode = "FULL_ACCESS"
  3. Report policies not using restricted access

Recommendation: Use restricted access mode with explicit allow-lists for outbound connections. Full access mode provides no protection against data exfiltration.

Cloud Support: AWS ‚úÖ | Azure ‚úÖ | GCP ‚ùå

NS-11: Network Policies Are Enforced (Not Dry-Run)

Severity: Medium | Check ID: 113 | Category: Network Security

Description: Validates that network policies are in enforced mode, not dry-run.

Logic:

  1. Query all account network policies
  2. Identify policies with enforcement_mode = "DRY_RUN"
  3. Report policies not actively enforcing restrictions

Recommendation: Move network policies from dry-run to enforced mode after validating that legitimate traffic is allowed. Dry-run mode logs violations but does not block them.

Cloud Support: AWS ‚úÖ | Azure ‚úÖ | GCP ‚ùå

NS-12: Network Policies Have Destination Allow-Lists

Severity: Medium | Check ID: 114 | Category: Network Security

Description: Validates that network policies in restricted mode have explicit destination allow-lists configured.

Logic:

  1. Query network policies with access_mode = "RESTRICTED"
  2. Check if allowed_destinations or allowed_fqdns fields are empty
  3. Report policies with no allow-list (deny-all configuration)

Recommendation: Define explicit allow-lists for required destinations (cloud storage, external APIs, etc.) when using restricted access mode. Empty allow-lists may block legitimate workload traffic.

Cloud Support: AWS ‚úÖ | Azure ‚úÖ | GCP ‚ùå

NS-13: Serverless SQL Warehouses Have Network Policy Coverage

Severity: High | Check ID: 115 | Category: Network Security

Description: Ensures serverless SQL warehouses are in workspaces with network policies.

Logic:

  1. Query SQL warehouses with enable_serverless_compute = true
  2. Cross-reference workspace_id with network policy assignments
  3. Report serverless warehouses without network policy coverage

Recommendation: Serverless SQL warehouses can access external data sources. Configure network policies to control outbound connections from these warehouses.

Cloud Support: AWS ‚úÖ | Azure ‚úÖ | GCP ‚ùå

INFO-18: Network Policy Default vs Custom Assignment

Severity: Low | Check ID: 116 | Category: Informational

Description: Identifies workspaces using default network policy vs custom policies for visibility.

Logic:

  1. Query network policy assignments for each workspace
  2. Identify workspaces using the account's default policy
  3. Report for visibility (informational only)

Recommendation: Default network policies apply to all workspaces without explicit assignment. Consider creating custom policies for workspaces with different security requirements.

Cloud Support: AWS ‚úÖ | Azure ‚úÖ | GCP ‚ùå

5. Implementation Example

Sample implementation for NS-10 (Network Policies Use Restricted Access Mode):

# In workspace_analysis.py

check_id = '112'  # NS-10
enabled, sbp_rec = getSecurityBestPracticeRecord(check_id, cloud_type)

def network_policy_restricted_mode_check(df):
    """Check if network policies use restricted access mode"""
    if df is not None and not isEmpty(df):
        violations = df.collect()
        violations_dict = {
            row.network_policy_id: [row.name, row.access_mode, row.workspace_count]
            for row in violations
        }
        return (check_id, 1, violations_dict)  # FAIL
    else:
        return (check_id, 0, {})  # PASS

if enabled:
    sql = f'''
        SELECT
            network_policy_id,
            name,
            access_mode,
            SIZE(workspace_ids) as workspace_count
        FROM {intermediate_schema}.account_networkpolicies
        WHERE access_mode = 'FULL_ACCESS'
    '''
    sqlctrl(workspace_id, sql, network_policy_restricted_mode_check)

Potential Challenges

  1. Workspace-to-Policy Mapping: If no direct API exists for workspace bindings, derive from:
    • Network policy objects (may contain workspace_ids array)
    • Workspace metadata (may contain network_policy_id field)
    • Cross-referencing tables in SQL analysis logic
  2. Default Policy Detection: Need to identify which policy is the account default
    • Check for is_default: true flag in API response
    • Or identify by name convention
  3. Policy Structure Variability: Field names may vary from documentation
    • Handle null/missing fields gracefully
    • Different clouds may return different structures (especially Azure)

The implementation of serverless egress control checks fills a critical security gap in SAT. While the tool already collects network policy data, it currently performs no analysis on this data.
